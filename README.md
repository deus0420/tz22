### Анализ процесса и выявленные противоречия
На основе описания "как есть" были выявлены следующие проблемы:

1. **Избыточность систем:** Использование трех независимых систем (№1, №2, №3) приводит к двойной работе. Техподдержка вынуждена закрывать одну и ту же заявку дважды вручную.
2. **Отсутствие контроля:** В описании отсутствует этап согласования закупки. Снабжение заказывает оборудование у поставщика без подтверждения бюджета.
3. **Информационный разрыв:** Пользователь видит номер обращения в Системе №1, но процесс закупки происходит в Системе №2 и №3. При отсутствии интеграции пользователь не получает актуальный статус "Ждем поставку".
4. **Непрозрачность логистики:** Вызов логистики описан как ручное действие, которое не фиксируется в системах, что может привести к потере оборудования при передаче.

## Задача 2: Разработка функциональности публикации товара

### 1. User Story (Верхнеуровневое описание)
**Как** Продавец Личного кабинета,  
**Я хочу** иметь возможность самостоятельно создавать и настраивать карточку товара,  
**Чтобы** представить свой ассортимент на витрине маркетплейса и начать продажи.

**Критерии приемки (Acceptance Criteria):**
* Наличие формы с обязательными полями (Наименование, Категория, Цена, Описание).
* Возможность загрузки медиа-контента (фото/видео).
* Валидация данных перед отправкой (проверка на пустые поля и формат цены).
* Автоматический перевод товара в статус "На модерации" после сохранения.

---

### 2. Use Case (Вариант использования)

**Название:** Публикация нового товара в ЛК Продавца.  
**Актер:** Продавец.  
**Предусловие:** Продавец успешно авторизован, профиль магазина заполнен и активен.

| № | Действие Актера | Ответ Системы |
|:---|:---|:---|
| 1 | Нажимает кнопку «Добавить товар» в разделе «Товары» | Отображает форму создания товара с пустыми полями |
| 2 | Выбирает категорию товара из выпадающего списка | Подгружает специфические атрибуты для выбранной категории |
| 3 | Заполняет основные поля (Имя, Описание, Артикул) и указывает цену | Проверяет корректность ввода (цена > 0, наличие символов) |
| 4 | Загружает фотографии товара | Выполняет проверку формата (PNG/JPG) и размера (до 5 Мб) |
| 5 | Нажимает кнопку «Опубликовать» | Проводит финальную валидацию и сохраняет объект в БД |
| 6 | — | Присваивает товару статус `Pending` (На модерации) и выводит сообщение об успехе |

**Альтернативные сценарии:**
* **3.1 Ошибка валидации:** Если цена не указана, поле подсвечивается красным, кнопка «Опубликовать» заблокирована.
* **4.1 Некорректный файл:** Если загружен формат .pdf, система выводит системное сообщение: «Неподдерживаемый формат файла».

---

### 3. Логическая схема процесса (As Is)
Процесс публикации включает следующие ключевые этапы:
1. **Заполнение данных** (User Input).
2. **Валидация** (Системная проверка на стороне Front-end и Back-end).
3. **Модерация** (Бизнес-проверка на соответствие правилам площадки).
4. **Публикация** (Отображение товара на витрине для покупателей).

> **Примечание аналитика:** Для минимизации отказов на этапе модерации рекомендуется внедрить подсказки (tooltips) для заполнения полей и автоматическую проверку качества изображений при загрузке.

## Задача 3: Проектирование REST API и алгоритма регистрации

### 3.1. Спецификация API метода регистрации

**Метод:** `POST`  
**URL:** `/api/v1/auth/register`  
**Content-Type:** `application/json`

#### Входные параметры (Request Body)
| Параметр | Тип | Обяз. | Ограничения | Описание |
|:---|:---|:---:|:---|:---|
| `firstName` | String | Да | 1-50 символов | Имя пользователя |
| `lastName` | String | Да | 1-50 символов | Фамилия пользователя |
| `username` | String | Да | Уникальное значение | Логин для входа |
| `password` | String | Да | Мин. 8 симв, цифра, заглавная, спецсимвол | Пароль |
| `captchaToken` | String | Да | Валидный токен | Результат прохождения ReCaptcha |

#### Выходные параметры (Success 201 Created)
| Параметр | Тип | Обяз. | Описание |
|:---|:---|:---:|:---|
| `userId` | UUID/Int | Да | Уникальный ID созданного пользователя |
| `message` | String | Да | Текст: "User Register Successfully" |

#### Ошибки (Error Responses)
| Код | Описание | Сообщение (message) |
|:---|:---|:---|
| `400` | Ошибка валидации / Капча | "Passwords must have at least one alphanumeric character..." / "Please verify reCAPTCHA" |
| `409` | Конфликт данных | "User exists!" (Логин уже занят) |
| `500` | Ошибка сервера | "Internal Server Error" |

### 3.2. Алгоритм создания пользователя (Back-end logic)

При получении запроса на регистрацию сервис выполняет следующие шаги:

1. **Валидация формата данных:** Проверка, что все обязательные поля заполнены и соответствуют типам данных.
2. **Проверка пароля:** Проверка на соответствие требованиям безопасности (длина, спецсимволы). Если не проходит — возврат `400 Bad Request`.
3. **Верификация ReCaptcha:** Отправка полученного `captchaToken` во внешний сервис Google для подтверждения, что запрос пришел от человека. Если токен невалиден — возврат `400`.
4. **Проверка на уникальность (Idempotency):** Поиск в БД пользователя с таким же `username`. Если найден — возврат `409 Conflict`.
5. **Хеширование пароля:** Генерация соли и хеширование пароля (например, через BCrypt). *Никогда не сохраняем пароль в чистом виде!*
6. **Запись в БД:** Создание новой записи в таблице `Users`.
7. **Формирование ответа:** Возврат успешного статуса `201` и ID пользователя.
